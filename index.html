安卓开发那些事_004_数据存储之SQLite

发表于 2016-03-04   |   分类于 安卓开发那些事   |  

版权声明：本文为博主原创文章，未经博主允许不得转载。

SQLite的属性特点注定了他非常适合运用在移动端，在android开发中如果有需求用到数据库时自己本就可以实现了，API都是现成的。但是碰到数据量非常庞大的时候就必须考虑数据库的相关优化问题。读取写入时间，性能等等。如果数据是一些必须要保证不被第三方恶意获取的话还要考虑到数据库的加密问题。接下来就说说一些开源的数据库框架，以及数据库加密相关

Android开源ORM（对象关系数据映射）

你优化SQLite的功力可能不是很强，但是有大牛的肩膀可以依靠，github上开源的orm框架其实非常多，第一的肯定是greendao。顶顶大名，如雷贯耳，包括他的eventbus，笔者只有膜拜的份

网上也有一些对greendao与其他orm框架的实际测试对比，greendao也是独占鳌头，赢得不要不要的，有兴趣的朋友可以去看看

贴下地址：https://github.com/greenrobot/greenDAO

上面介绍的很详尽，用法什么的这里不讲了，框架中有一个generator的java工程为了生存javabean和DaoMaster类，并未使用annotation方式，也是为了性能，不过我个人感觉如果在完美一点可以推出一个android studio的插件自动生存工具，这就完美了

对于SQLite优化中，使用sql语句速度会快一些，善用session提高性能等这些都可以在源码中看到，学习优化数据库的童鞋，这个工程也是一个学习的途径

使用过程中遇到一点不爽的地方就是如果你需要在添加一张表，你必须要用generator的java工程再次生成一遍，倒也不是慢，也不是很麻烦，总感觉会有更好的方法来避免这个问题，对于我这种懒人还是有点矫情

Android中SQLite加密

还是一个github工程，叫做SQLCipher，地址：
https://github.com/sqlcipher/android-database-sqlcipher

感觉我真的是github的搬运工…有辣么句话，我们不生产代码，我们只是github的搬运工…

SQLite本身是有加密接口的，只是android没有封装api，当然我指的是加密数据库，不是对数据加密存储。你自己也可以去实现接口然后build一个加密的数据库出来，花个一周时间应该可以接受，当然你也可以运用我给你推荐的这个库，先用用，不满意的地方自己在完善一下，而且人家还支持数据加密，完美了，也是一周时间…

这个工程有一个非常好的地方，就是保留了所有android的自身API，如果你要替换一个数据库，直接将包换一下，在onCreat初始化一下SQLCipher就OK了，这也使得你可以拥有一个快速、高效、并且加密的SQLite了

android开发中离不开数据的持久化，存在服务器端、写到文件、sharedpreference、sqlite是android处理数据存储的几种常见方式，下面介绍一下sharedpreference。sqlite存储将在下篇介绍

android机器root的简单便捷使得android在存储数据时难免的为之担心，而存在服务器有时并不方便，所以一些加密的数据存储方法应运而生

secure preference

preference存储的是键值对形式的.xml文件，如果root了机器，这个文件存储的数据就是明文存在的了，为了避免这个情况发生，我们只能从存入的键和值入手解决问题

下面贴两个github的开源项目

https://github.com/scottyab/secure-preferences

https://github.com/ophio/secure-preferences

这两个是github上面排名暂时第一第二的两个项目，比较一下两者中的核心代码都是用的android自带的对称加密算法对键值对加密，一个AES一个DES的加密，再有不同的地方就是对自定义的sharedpreference的封装方法

如果对安全要求在高一些，加密可以放到native之中来做

ACRA

ACRA（application crash report for android)，是google的一个开源项目，引入它的包，可以将crash log发送到自己的google文档，官方文档有详细说明。http://code.google.com/p/acra/

该库已经对crash的上报信息进行了整合，包括手机的一些基本信息，使用上也可以高度定制自己需要的一些字段，上报的地址，也可以实现ReportSender方法来定制report方法，包括发送邮件、http到server、googleform。当然你也可以自定义到文件。具体实现谷哥度娘一大堆，这里不贴了。

说明一点，log和crash如果存在手机的文件中，要考虑一下上报到服务器的时间和手机状态（网络、电量等）

Bugly

国内较出名的crash report工具，企鹅出品，但貌似用的多的还是自家的app…

bugly涵盖了java层的crash收集、ANR的收集、native层的crash收集、30kb的Log缓存上报以及crash数据分析等功能，具体可到官网查看。功能还是比较强大的，对于有native开发的app来说还是有用处的，但用处多少，这里不好说，因为自身开发native的话一定会有一套自己的检测机制。

bugly的AS集成在开发者使用手册中写的已经很详尽了，贴一下地址：http://bugly.qq.com/androidfast

Android Crash Report那些事

在开发中，会遇到crash问题，一般来说，crash发生在java层，但是，有时候会发生在其他层面上。大致，Android Crash 大致有三类：

  • Java uncatch exception
  • ANR crash
  • Native crash

本文主要针对Java uncatch exception来表，ANR在开发中还是比较容易避免的，Native Crash如果有兴趣的朋友可以自行在深入研究一下

ProGuard相关

When optimizing your application with ProGuard, include the following rules to prevent logback-android and SLF4J calls from being removed (unless that’s desired):

-keep class ch.qos.** { *; }

-keep class org.slf4j.** { *; }

-keepattributes *Annotation*

If you don’t use the mailing features of logback (i.e., the SMTPAppender), you might encounter an error while exporting your app with ProGuard. To resolve this, add the following rule:

-dontwarn ch.qos.logback.core.net.*
